# Верификация email и сброс пароля

**Цель:** разобраться, как в `apps/api` устроены подтверждение почты и сброс пароля на одноразовых токенах, как настроить отправку писем и как это работает в dev без реального SMTP.

Архитектурные решения за этим флоу зафиксированы в [ADR-011](../adr/011-email-flow.md).

## Предварительные требования

- Поднятая БД и применённые миграции (`pnpm --filter @repo/api db:migrate`) — флоу использует таблицу `VerificationToken` и поле `User.emailVerified`.
- Заданные env-переменные mailer-блока (см. ниже). В dev достаточно дефолтов.

## Переменные окружения

Все переменные живут в корневом `.env` (см. `.env.example`).

| Переменная               | Назначение                                                                        | Default                 |
| ------------------------ | --------------------------------------------------------------------------------- | ----------------------- |
| `MAIL_TRANSPORT`         | `json` — dev (письма логируются, SMTP не нужен); `smtp` — реальная отправка       | `json`                  |
| `MAIL_FROM`              | Адрес в поле From исходящих писем                                                 | `no-reply@example.com`  |
| `SMTP_HOST`              | Хост SMTP-сервера (**обязателен** при `MAIL_TRANSPORT=smtp`)                      | —                       |
| `SMTP_PORT`              | Порт SMTP; `465` → implicit TLS, остальные → STARTTLS (**обязателен** при `smtp`) | —                       |
| `SMTP_USER`              | Логин SMTP (**обязателен** при `smtp`)                                            | —                       |
| `SMTP_PASS`              | Пароль SMTP (**обязателен** при `smtp`)                                           | —                       |
| `WEB_URL`                | Базовый URL веб-приложения для ссылок в письмах                                   | `http://localhost:3000` |
| `EMAIL_VERIFICATION_TTL` | Срок жизни токена верификации, формат `<число><s\|m\|h\|d\|w>`                    | `24h`                   |
| `PASSWORD_RESET_TTL`     | Срок жизни токена сброса пароля, формат `<число><s\|m\|h\|d\|w>`                  | `1h`                    |

> При `MAIL_TRANSPORT=smtp` все поля `SMTP_*` обязательны — иначе приложение упадёт на старте (валидация env через zod `superRefine`).
> Формат TTL валидируется строго (`^\d+[smhdw]$`): опечатка вроде `24hh` тоже валит старт, чтобы не сократить срок токена молча.

## Как работает dev-режим (jsonTransport)

При `MAIL_TRANSPORT=json` (default) `MailerService` использует `nodemailer.jsonTransport`: письмо не уходит наружу, а сериализуется в JSON и логируется через pino:

```
LOG [MailerService] { msg: 'Email sent (jsonTransport)', message: '{...тело письма со ссылкой...}' }
```

Ссылку для верификации/сброса можно скопировать прямо из лога — реальный SMTP в разработке и тестах не нужен. В e2e-тестах токен извлекается из перехваченного письма или напрямую из таблицы `VerificationToken`.

## Эндпоинты

Все находятся в `apps/api/src/auth/auth.controller.ts`, документированы в Swagger (`/api/docs`).

| Метод и путь                     | Назначение                                                                                                              | Ответ          |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------- | -------------- |
| `POST /auth/register`            | Регистрация: создаёт пользователя, выпускает токен верификации и шлёт письмо. Логин разрешён сразу (возвращает токены). | `201` + токены |
| `POST /auth/verify-email`        | Подтверждение почты по одноразовому токену (`{ token }`).                                                               | `204`          |
| `POST /auth/resend-verification` | Повторная отправка письма верификации (`{ email }`). Тихий no-op, если email не найден или уже подтверждён.             | `204`          |
| `POST /auth/forgot-password`     | Запрос сброса пароля (`{ email }`). Ответ одинаков независимо от существования email.                                   | `204`          |
| `POST /auth/reset-password`      | Установка нового пароля по токену (`{ token, password }`). Сбрасывает все сессии пользователя.                          | `204`          |

### Флоу верификации

1. `POST /auth/register` → создаётся `User` (`emailVerified: false`), выпускается `EMAIL_VERIFICATION`-токен, на почту уходит ссылка `${WEB_URL}/verify-email?token=...`.
2. Пользователь переходит по ссылке, фронт вызывает `POST /auth/verify-email` с токеном.
3. Токен проверяется (хэш + TTL), удаляется (одноразовость), `User.emailVerified` становится `true`.
4. Профиль (`GET /users/me`) теперь возвращает `emailVerified: true`.

Неподтверждённый пользователь логинится и работает, но маршруты под `VerifiedGuard` (`apps/api/src/auth/guards/verified.guard.ts`) отдают `403 EmailNotVerified`. Чтобы закрыть маршрут верификацией, навесьте гвард поверх `JwtAuthGuard`:

```ts
@UseGuards(JwtAuthGuard, VerifiedGuard)
```

### Флоу сброса пароля

1. `POST /auth/forgot-password` с email. Если пользователь существует — выпускается `PASSWORD_RESET`-токен и уходит письмо со ссылкой `${WEB_URL}/reset-password?token=...`. Ответ всегда `204`, чтобы не раскрывать существование учётки.
2. Пользователь переходит по ссылке, фронт вызывает `POST /auth/reset-password` с токеном и новым паролем (минимум 8 символов).
3. Токен проверяется и удаляется, пароль перехэшируется (bcrypt), `refreshToken` обнуляется — все активные сессии разлогиниваются.

## Безопасность

- В БД хранится только `sha256` токена — утечка таблицы не раскрывает рабочие токены.
- Токены одноразовые (удаляются при `consume`) и с TTL; выпуск нового токена того же типа инвалидирует старые.
- `forgot-password` отвечает одинаково для существующего и несуществующего email.

## Аудит

Флоу пишет в аудит-журнал события `EMAIL_VERIFIED`, `PASSWORD_RESET_REQUESTED`, `PASSWORD_RESET_COMPLETED` через декоратор `@Audit(...)` (см. [гайд по аудиту](./adding-audit-event.md)). `forgot-password` логирует в metadata только email и не раскрывает, существует ли он.

## Проверка

```bash
# unit-тесты mailer/verification/auth
pnpm --filter @repo/api test

# сквозной e2e-флоу (регистрация → verify, forgot → reset); требует Postgres
pnpm --filter @repo/api test:e2e
```

Ручная проверка в dev: зарегистрируйтесь через Swagger (`/api/docs`), скопируйте ссылку из лога `MailerService` и вызовите `POST /auth/verify-email` с извлечённым токеном — `GET /users/me` должен показать `emailVerified: true`.

## Частые проблемы

| Проблема                                             | Причина                                                                             | Решение                                                                |
| ---------------------------------------------------- | ----------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| Приложение падает на старте с ошибкой по `SMTP_*`    | `MAIL_TRANSPORT=smtp`, но поля SMTP не заданы                                       | Заполнить `SMTP_HOST/PORT/USER/PASS` или вернуть `MAIL_TRANSPORT=json` |
| Старт падает с ошибкой формата TTL                   | `EMAIL_VERIFICATION_TTL`/`PASSWORD_RESET_TTL` не в формате `<число><s\|m\|h\|d\|w>` | Привести к виду `24h`, `1h`, `30m` и т.д.                              |
| `verify-email`/`reset-password` отдаёт ошибку токена | Токен истёк, уже использован или инвалидирован новым выпуском                       | Запросить новое письмо (`resend-verification` / `forgot-password`)     |
| Письма не приходят на почту в dev                    | Включён `MAIL_TRANSPORT=json` — это ожидаемо                                        | Ссылку взять из лога `MailerService`; для реальной доставки — `smtp`   |
| `403 EmailNotVerified` на защищённом маршруте        | Email пользователя ещё не подтверждён                                               | Пройти верификацию или снять `VerifiedGuard` с маршрута                |
