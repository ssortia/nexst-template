# UI: курсор pointer при наведении на кликабельные элементы (#27)

## Overview

- Обеспечить, чтобы при наведении на интерактивные элементы (кнопки, ссылки, кастомные кликабельные элементы) курсор менялся на `pointer`, а на disabled-состояниях — на `not-allowed`.
- Проблема: нативная `<button>` по умолчанию показывает обычный курсор (не pointer), из-за чего пользователь не всегда понимает, что элемент кликабелен. Это ухудшает UX.
- Решение интегрируется через глобальное правило в `@layer base` (`globals.css`), что даёт единый источник истины и автопокрытие будущих элементов. Кастомные `onClick`-элементы, не являющиеся нативными `<button>`/`<a>`, проверяются и при необходимости получают явный класс.

## Solution Overview

- **Главный механизм:** добавить в `@layer base` в `globals.css` правила:
  - `button:not(:disabled)` и `[role="button"]:not([aria-disabled="true"])` → `cursor: pointer`
  - `button:disabled` и `[role="button"][aria-disabled="true"]` → `cursor: not-allowed`
- Селектор сознательно ограничен только `button` и `[role="button"]` — никаких `summary`/`label[for]` (в проекте таких кликабельных кейсов нет; добавим только если появятся — YAGNI).
- **Все** `<button>` намеренно попадают под `cursor: pointer` — это осознанное решение (в т.ч. `link`-вариант `Button`, который рендерится как `<button>`), а не побочный эффект широкого селектора.
- Это покрывает все нативные `<button>` (включая `Button` из shadcn и кнопки пагинации) и элементы с `role="button"` единообразно, без правки каждого компонента.
- Нативные `<a>`/`<Link>` уже показывают pointer по умолчанию — отдельных правил не требуют.
- Кастомные `onClick`-элементы на не-кнопках (`<th>`, `<div>` и т.п.) глобальное правило по тегу не покрывает — для них проводится аудит и при необходимости добавляется `cursor-pointer` (большинство уже имеет его).

### Решение по disabled-курсору (важно)

- У shadcn `Button` стоит `disabled:pointer-events-none`. При `pointer-events: none` браузер **не** пересчитывает курсор над элементом, поэтому глобальное `button:disabled { cursor: not-allowed }` на shadcn `Button` **не сработает** — disabled-кнопка сохранит дефолтный курсор.
- **Принятое решение:** `button.tsx` не трогаем, `pointer-events-none` оставляем. Клики по disabled-кнопке и так блокируются атрибутом `disabled`, а `opacity-50` визуально сигнализирует неактивность — отдельный `not-allowed` для shadcn `Button` не требуется.
- `cursor: not-allowed` гарантируется для нативных disabled-кнопок **без** `pointer-events-none` — например, кнопки пагинации audit-table, у которых уже есть точечный `disabled:cursor-not-allowed` (оставляем как есть), плюс глобальное правило как страховка для будущих таких кнопок.
- Критерий приёмки соответственно уточняется в Task 3 (см. ниже).

## Technical Details

- Правило размещается в существующем блоке `@layer base` в `apps/web/src/app/globals.css`, рядом с `*` и `body`.
- См. раздел «Решение по disabled-курсору» в Solution Overview: `button.tsx` не меняем, disabled shadcn `Button` сохраняет дефолтный курсор из-за `pointer-events-none`; `not-allowed` обеспечивается для нативных кнопок без `pointer-events-none` (пагинация уже имеет точечный `disabled:cursor-not-allowed`).
- Глобальный селектор ограничен только `button` и `[role="button"]`, чтобы не задеть нежелательные элементы.
