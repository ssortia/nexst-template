# Аудит-лог действий пользователей (issue #23)

## Overview

Добавить сквозной журнал аудита значимых действий пользователей, покрывающий все слои монорепо:

- **API** — автоматическая фиксация действий через декларативный декоратор `@Audit(...)` + глобальный `AuditInterceptor`, без ручных вызовов в бизнес-логике.
- **Хранение** — отдельная таблица `audit_logs` в Prisma с индексами под типовые фильтры.
- **Типы** — Zod-схемы события аудита в `@repo/types` как единый source of truth для API и web.
- **Web** — страница `/admin/audit` с серверной фильтрацией (актор / тип события / диапазон дат), сортировкой по дате и пагинацией, по образцу существующей таблицы пользователей.

**Проблема, которую решает:** сейчас невозможно узнать, кто и когда выполнял значимые действия (логин, смена роли и т.п.). Для production-ready основы такой журнал нужен для безопасности и разбора инцидентов.

**Интеграция:** новый модуль `audit` встраивается рядом с `auth`/`users`, переиспользует слой репозитория (ADR-007), RBAC-гарды (`JwtAuthGuard` + `RolesGuard`), паттерн API-клиента фронта (ADR-008) и паттерн серверной таблицы из `admin/users`.

## Solution Overview

**Механизм (выбран):** декларативный декоратор `@Audit(options)` на методах контроллеров + один глобальный `AuditInterceptor`, читающий метаданные через `Reflector`.

Поток:

```
HTTP-запрос
  -> Guard (JwtAuthGuard) проставляет req.user
  -> AuditInterceptor: читает @Audit-метаданные хендлера
       success -> tap(): пишет событие options.event (success=true)
       error   -> catchError(): пишет options.failureEvent ?? options.event (success=false), затем rethrow
  -> AuditService.record() -> AuditRepository.create()  (fire-and-forget, ошибки аудита не ломают запрос)
```

Ключевые решения и обоснование:

1. **Декоратор + interceptor**, а не event-based: фиксация остаётся декларативной и рядом с эндпоинтом, бизнес-логика не засоряется ручными `emit`-вызовами (требование issue), неуспешный логин ловится через `catchError` без явного кода в сервисе.
2. **Whitelist метаданных**: interceptor НИКОГДА не логирует тело запроса целиком. В лог попадают только поля, явно возвращённые резолверами `target`/`metadata`/`actor` из декоратора. Это гарантирует, что пароли/токены не утекут в журнал.
3. **`actorId` без FK-связи на `User`**: храним `actorId`/`targetId` как обычные строки + денормализованный `actorEmail`. Журнал должен переживать удаление пользователя, поэтому каскадных связей и `onDelete` нет.
4. **`metadata` как `Json`**: структура события расширяема без миграций — легко добавлять новые типы событий (требование issue).
5. **Доступ к журналу — только ADMIN** (`JwtAuthGuard` + `RolesGuard` + `@Roles(ADMIN)` на API; redirect не-ADMIN в `admin/layout.tsx` на web).
6. **Удаление пользователя** — добавляется только тип события `USER_DELETED` в перечислении и инфраструктура; сам DELETE-эндпоинт в этой задаче НЕ реализуется (отдельная задача), декоратор подключится к нему позже.

## Technical Details

### Prisma — модель и enum

```prisma
enum AuditEvent {
  LOGIN_SUCCESS
  LOGIN_FAILED
  LOGOUT
  USER_ROLE_CHANGED
  USER_DELETED        // тип заведён заранее; эндпоинт удаления — в будущей задаче
}

model AuditLog {
  id         String     @id @default(cuid())
  event      AuditEvent
  success    Boolean    @default(true)
  actorId    String?    // id актора; null для неуспешного логина. Без FK — лог переживает удаление user
  actorEmail String?    // денормализованный email актора для отображения без join
  targetId   String?    // id целевой сущности (напр. пользователь, чью роль меняют)
  targetType String?    // тип целевой сущности ("User")
  metadata   Json?      // структурированные доп. данные (только whitelisted поля)
  ip         String?
  userAgent  String?
  createdAt  DateTime   @default(now())

  @@index([createdAt])
  @@index([event])
  @@index([actorId])
  @@map("audit_logs")
}
```

### `@repo/types` — схемы (новый файл `audit.ts`)

- `AuditEventSchema = z.enum([...])`, `AuditEvent`.
- `AuditLogSchema` (id, event, success, actorId/Email, targetId/Type, metadata, ip, userAgent, createdAt: `z.coerce.date()`).
- `ListAuditQuerySchema` / `ListAuditParams`: `actor?`, `event?`, `dateFrom?`, `dateTo?`, `sortOrder?` (`asc`/`desc` по `createdAt`), `limit?`, `offset?`.

- `AUDIT_SORTABLE_FIELDS = ['createdAt'] as const` (единый источник истины).

### API — декоратор и опции

⚠️ Типизация request: в проекте нет готового augmented-типа `FastifyRequest` с `user`. Passport (`jwt.strategy.validate()`) кладёт в `req.user` **полный Prisma-`User`** (не slim `{id,email}`). Под strict mode голый `FastifyRequest` не даёт доступа к `user`/`body`/`params`, поэтому в декораторе/interceptor вводим локальный тип:

```ts
import type { User } from '@prisma/client';
import type { FastifyRequest } from 'fastify';

// Локальный тип запроса для резолверов аудита.
type AuditRequest = FastifyRequest & {
  user?: User; // полный User от JwtAuthGuard (Passport)
  body?: Record<string, unknown>; // тело запроса (резолверы берут только whitelisted поля)
  params?: Record<string, string>; // route-параметры
};

export interface AuditOptions {
  event: AuditEvent; // событие при успехе
  failureEvent?: AuditEvent; // событие при ошибке (напр. LOGIN_FAILED)
  targetType?: string; // "User"
  target?: (req: AuditRequest) => string | undefined; // id целевой сущности
  actor?: (req: AuditRequest) => { id?: string; email?: string } | undefined; // переопределение актора
  metadata?: (req: AuditRequest) => Record<string, unknown> | undefined; // только безопасные поля
}
export const AUDIT_KEY = 'audit:options';
export const Audit = (options: AuditOptions) => SetMetadata(AUDIT_KEY, options);
```

Дефолт актора в interceptor: `req.user` (читаем `user.id`/`user.email` из полного `User`), проставленный `JwtAuthGuard`. Для login (без guard) `req.user` отсутствует — актор берётся из `options.actor` → `req.body.email`. IP — `req.ip`, UA — `req.headers['user-agent']`.

### API — разметка декораторами

- `POST /auth/login`: `@Audit({ event: LOGIN_SUCCESS, failureEvent: LOGIN_FAILED, actor: (r) => ({ email: r.body?.email }), metadata: (r) => ({ email: r.body?.email }) })` — пароль не логируется.
- `POST /auth/logout`: `@Audit({ event: LOGOUT })` (актор из `req.user`).
- `PATCH /users/:id/role`: `@Audit({ event: USER_ROLE_CHANGED, targetType: 'User', target: (r) => r.params.id, metadata: (r) => ({ role: r.body?.role }) })`.

### API — репозиторий (контракт BaseRepository)

⚠️ Важно: `BaseRepository` (`apps/api/src/common/repository/base.repository.ts`) даёт только CRUD-по-id через структурный `BaseModelDelegate` — у него **нет** `count` и `$transaction`, а `create` объявлен `protected`. Нетривиальные запросы делаются как в `UsersRepository`: инъекция `PrismaService` напрямую и обращение к типизированному делегату `this.prisma.auditLog.*`.

Поэтому `AuditRepository`:

- `extends BaseRepository<AuditLog, Prisma.AuditLogCreateInput, Prisma.AuditLogUpdateInput>` с кастом делегата `prisma.auditLog as unknown as BaseModelDelegate<...>` (как в `users.repository.ts`);
- `record()` — публичный метод-обёртка: `override record`/`create` поверх `super.create(...)`, чтобы вызывать из `AuditService` (по образцу `override create` в `UsersRepository`);
- `findPage()` — НЕ через базовый делегат: `this.prisma.$transaction([this.prisma.auditLog.findMany({...}), this.prisma.auditLog.count({ where })])`.

`findPage` — `where`: `event` (eq), `actor` (по `actorEmail contains insensitive` ИЛИ `actorId eq`), `createdAt` (`gte dateFrom`, `lte dateTo`). `orderBy: { createdAt: sortOrder ?? 'desc' }`. `take: limit ?? 50` (cap 100), `skip: offset ?? 0`. Возврат `{ items, total }`.

### Web

- **Политика ретеншна/ротации**: при росте `audit_logs` добавить плановую очистку старых записей (cron/scheduled job) или партиционирование; вынести срок хранения в env (`AUDIT_RETENTION_DAYS`).

- **Подключение `USER_DELETED`**: повесить `@Audit({ event: USER_DELETED, ... })` на DELETE-эндпоинт пользователя, когда он появится (отдельная задача).
