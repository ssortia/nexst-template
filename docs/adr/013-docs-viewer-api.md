# ADR-013: Выдача документации (docs/) через API для просмотра в админке

**Статус:** Принято
**Дата:** 2026-06-15

---

## Контекст

Документация проекта (ADR, гайды, планы) живёт только в каталоге `docs/` репозитория. Администраторам удобно читать её прямо из веб-интерфейса (`/admin/docs`), без доступа к исходникам и без клонирования репозитория.

Ограничения: документация — потенциально чувствительная (внутренние решения, планы), поэтому доступ должен быть ограничен ролью `ADMIN`; чтение файлов с диска по пути из запроса открывает классический вектор path traversal; в проде api работает из скомпилированного `dist`, и каталог `docs/` по умолчанию в образ не попадает.

---

## Рассмотренные варианты

- **Источник: отдельный API-модуль vs статическая раздача каталога.** Статическая раздача (`@fastify/static`) не даёт навешать RBAC-гарды per-route и удобно сгруппировать дерево по подпапкам. Выбран отдельный NestJS-модуль `docs` — единый паттерн с прочими модулями (`audit`), guard-метаданные, Swagger, контракты через `@repo/types`.
- **Путь к docs: env-переменная vs детерминированный путь от `__dirname`.** Env добавляет точку отказа в проде (надо не забыть выставить) и усложняет конфиг. Путь разрешается детерминированно от `__dirname` (для тестируемости — внедряется через DI-токен `DOCS_ROOT`, чтобы тесты подставляли каталог-фикстуру и не зависели от `__dirname`).
- **Рендеринг Markdown: `react-markdown` без raw-HTML vs с `rehype-raw`.** `rehype-raw` пропускает сырой HTML из содержимого файла — XSS-вектор. Выбран `react-markdown` без `rehype-raw`: контент трактуется строго как Markdown.

---

## Решение

1. **Модуль `docs`** (`apps/api/src/docs/`) по образцу `audit`: `DocsService` + `DocsController` + `DocsModule`. Эндпоинты: `GET /docs` (дерево файлов, сгруппированное по подпапке первого уровня — `adr`/`guides`/`plans`, корень → группа `root`) и `GET /docs/file?path=...` (содержимое конкретного файла).

2. **Доступ только для ADMIN.** Оба эндпоинта под `@UseGuards(JwtAuthGuard, RolesGuard)` + `@Roles(Role.ADMIN)` + `@ApiBearerAuth()` — переиспользуется существующий RBAC (ADR-005).

3. **Защита от path traversal.** `getFile(relPath)` проверяет расширение `.md` и требует, чтобы `path.resolve(docsRoot, relPath)` начинался с `docsRoot + path.sep`. Это блокирует `..`, абсолютные пути (`/etc/passwd`), URL-encoded `..` (`%2e%2e`) и ложное совпадение префикса (вроде `docs-secret`). Невалидный путь → `BadRequestException`, отсутствующий файл → `NotFoundException` (нормализуются `AllExceptionsFilter`, ADR-012).

4. **Детерминированный путь к docs.** Корень внедряется провайдером через DI-токен `DOCS_ROOT` с дефолтом `path.resolve(__dirname, '../../../../docs')`. SWC даёт плоский `dist` (`apps/api/dist/docs/docs.service.js`), поэтому до корня монорепо ровно 4 уровня вверх: локально → `<repo>/docs`, в контейнере (`/app/apps/api/dist/docs/...`) → `/app/docs`. Внедрение через токен даёт тестируемость без зависимости от `__dirname`.

5. **Доставка в образ.** `docker/api.Dockerfile` копирует `docs/` в `/app/docs` (`COPY --from=builder /app/docs ./docs`) — целевой путь совпадает с дефолтом `DOCS_ROOT`.

6. **Контракты в `@repo/types`** (`DocsFileMeta`, `DocsTree`, `DocsFileContent`) — единый источник правды между API и web, исключает дрейф.

7. **Рендеринг на web** — `react-markdown` + `remark-gfm` без `rehype-raw`: raw-HTML запрещён, контент трактуется как Markdown.

---

## Последствия

### Положительные

- Документация читается из админки без доступа к исходникам — единый источник правды (файлы репозитория), а не копия в БД/CMS.
- Доступ ограничен ролью `ADMIN`, защита от path traversal покрыта unit-тестами (`../`, абсолютный путь, URL-encoded, ложный префикс).
- Контракты в `@repo/types` исключают расхождение API и web; рендеринг без raw-HTML закрывает XSS-вектор.

### Отрицательные / компромиссы

- Каталог `docs/` обязан попадать в Docker-образ — добавляется явный `COPY` в Dockerfile; забыли скопировать → эндпоинт вернёт пустое дерево.
- Отдаются только `.md`-файлы; прочие форматы (изображения, диаграммы) не поддерживаются.
- Только чтение: создавать или редактировать документацию из UI нельзя — это остаётся git-процессом.
