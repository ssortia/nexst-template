# ADR-011: Верификация email и сброс пароля на одноразовых токенах

**Статус:** Принято
**Дата:** 2026-06-09

---

## Контекст

В шаблоне не было проверки владения почтой при регистрации и не было механизма сброса пароля — двух базовых auth-сценариев, без которых production-ready основа неполна. Регистрация сразу логинила пользователя, никак не подтверждая, что email ему принадлежит. Забытый пароль восстановить было невозможно.

Требования: подтверждение почты и сброс пароля через одноразовые ссылки в письме; отправка писем не должна требовать реального SMTP в dev; механизм не должен раскрывать существование email на эндпоинте сброса; токены не должны храниться в открытом виде.

---

## Рассмотренные варианты

- **Отправка писем: nodemailer с транспортом по env vs внешний провайдер (SendGrid/SES SDK).** Внешний провайдер привязывает шаблон к конкретному вендору и требует учётных данных уже на этапе разработки. nodemailer — стандарт де-факто для Node, поддерживает `jsonTransport` (письмо сериализуется в JSON, не уходит наружу) для dev и обычный SMTP для prod, переключение — одной env-переменной.
- **Хранение токенов: отдельная таблица `VerificationToken` vs поля в `User`.** Поля в `User` (`verificationToken`, `resetToken`, `*ExpiresAt`) раздувают модель, не дают разных TTL и типов в одной структуре и плохо расширяются на новые типы токенов. Отдельная таблица с дискриминатором `type` — единый механизм для любых одноразовых токенов.
- **Политика неподтверждённого пользователя: блокировать логин vs логинить, но ограничивать.** Полный запрет логина до верификации усложняет UX (пользователь не может даже запросить повторное письмо из ЛК) и переусложняет поток. Выбран компромисс: логин разрешён, но защищённые верификацией маршруты закрыты гвардом.

---

## Решение

1. **Mailer-модуль (`apps/api/src/mailer/`)** поверх `nodemailer`. Транспорт выбирается по `MAIL_TRANSPORT`: `json` (dev, default) → `jsonTransport`, письмо логируется через pino (ссылку видно в логах/тестах); `smtp` → реальный SMTP из env (при `smtp` поля `SMTP_*` обязательны — zod `superRefine`). `MailerService` даёт типизированные методы `sendVerificationEmail` / `sendPasswordResetEmail`; ссылки строятся из `WEB_URL`.

2. **Отдельная таблица `VerificationToken`** с дискриминатором `type` (`EMAIL_VERIFICATION` | `PASSWORD_RESET`), `tokenHash @unique`, `expiresAt`, relation на `User` с `onDelete: Cascade`. Единый механизм для разных типов одноразовых токенов.

3. **Хэширование + одноразовость + TTL.** Пользователю отдаётся `randomBytes(32).toString('hex')`, в БД хранится только `sha256(token)` — утечка БД не раскрывает рабочие токены. При выпуске нового токена старые того же типа инвалидируются. `consume(token, type)` проверяет TTL, удаляет запись (одноразовость) и возвращает `userId`. TTL берётся из env (`EMAIL_VERIFICATION_TTL` default `24h`, `PASSWORD_RESET_TTL` default `1h`) и строго валидируется zod-regex `^\d+[smhdw]$` на старте — опечатка падает громко, а не молча сокращает срок токена.

4. **Политика `VerifiedGuard`.** Поле `User.emailVerified` (`Boolean @default(false)`) отражается в `PUBLIC_SELECT` и профиле. `VerifiedGuard` (поверх `JwtAuthGuard`) возвращает 403 `EmailNotVerified` для маршрутов, требующих подтверждённой почты. Логин при этом разрешён — неподтверждённый пользователь работает с ограничениями, а не заблокирован полностью.

5. **Единый ответ `forgot-password`.** Эндпоинт всегда отвечает одинаково (204) независимо от того, существует email или нет — чтобы не было оракула для перечисления учёток. `reset-password` сбрасывает `refreshToken` пользователя (разлогин всех сессий).

6. **Аудит.** Новые события `EMAIL_VERIFIED`, `PASSWORD_RESET_REQUESTED`, `PASSWORD_RESET_COMPLETED` логируются декоратором `@Audit(...)` (см. [ADR-010](./010-audit-log.md)); `forgot-password` пишет в metadata только whitelisted email и не раскрывает существование адреса.

Эндпоинты: `POST /auth/register` (выпускает verification-токен и шлёт письмо, логин сохраняется), `POST /auth/verify-email`, `POST /auth/resend-verification`, `POST /auth/forgot-password`, `POST /auth/reset-password`.

## Последствия

### Положительные

- Dev не требует реального SMTP — письма видны в логах, ссылка извлекается в тестах.
- Единая таблица токенов расширяется на новые типы без изменения схемы модели `User`.
- Утечка БД не раскрывает рабочие токены (хранится только хэш); токены одноразовые и с TTL.
- `forgot-password` не даёт оракула существования учётки.

### Отрицательные / компромиссы

- Неподтверждённый пользователь всё же получает сессию — защита переносится на `VerifiedGuard` на каждом чувствительном маршруте (надо не забыть навесить гвард).
- Значения `VerificationTokenType` и новых `AuditEvent` дублируются между Prisma-enum и Zod-схемами `@repo/types` — сверяются вручную.

### Риски

- Просроченные `VerificationToken` не подчищаются автоматически — при большом объёме потребуется cron-очистка (вне рамок этого решения).
- `jsonTransport` в dev не проверяет реальную доставку — корректность SMTP-настроек проверяется только вручную в prod-подобном окружении.
