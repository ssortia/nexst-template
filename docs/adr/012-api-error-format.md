# ADR-012: Единый формат тела ошибки API

**Статус:** Принято
**Дата:** 2026-06-10

---

## Контекст

Источники ошибок в `apps/api` были не согласованы: `HttpException` отдавал `{ statusCode, message, error }`, `ValidationPipe` клал `message` массивом строк, ошибки Prisma и неизвестные сбои улетали сырым 500 с потенциальной утечкой stack/SQL в тело. На фронте `ApiError.message` был всем сырым телом ответа (JSON-блобом), из-за чего формы хардкодили строки по `status` и не могли показать серверное сообщение.

Требования: предсказуемый shape тела для любой ошибки; единый человекочитаемый `message`; отсутствие утечки внутренних деталей в prod; обратная совместимость по статус-кодам существующих эндпоинтов.

---

## Рассмотренные варианты

- **Формат тела: минимальный `{ statusCode, message, details? }` vs RFC 7807 (`application/problem+json`).** RFC 7807 (`type`/`title`/`instance`) избыточен для шаблона: требует словаря типов ошибок и URI, усложняет фронт. Выбран минимальный shape — одна строка `message` плюс опциональный плоский `details[]` для сообщений валидации.
- **Валидация: per-field ошибки + Zod-фабрика vs плоский `details[]`.** Поле-уровневые ошибки дублируют client-side валидацию (`react-hook-form + zod`) и тянут кастомный `exceptionFactory`. Отдаём generic message + плоский массив сообщений из дефолтного `BadRequestException` — без factory, без привязки полей.
- **Нормализация: глобальный `ExceptionFilter` vs обработка в каждом сервисе/контроллере.** Точечная обработка не покрывает неизвестные сбои и ошибки Prisma из глубины. Глобальный `@Catch()`-фильтр — единая точка-safety net поверх явных исключений в коде.

---

## Решение

1. **Общий тип `ApiErrorBody`** (`packages/types/src/api-error.ts`, обычный TS-интерфейс без Zod) — единственный источник правды о форме тела: `{ statusCode: number; message: string; details?: string[] }`. Импортируется и API, и web.

2. **Глобальный `AllExceptionsFilter`** (`apps/api/src/common/filters/all-exceptions.filter.ts`, `@Catch()`), зарегистрирован в `main.ts` через `app.useGlobalFilters`. Пишет ответ через Fastify `reply` с guard на `reply.sent`. Маппинг:
   - `HttpException` → `getStatus()` + нормализованный строковый `message`;
   - валидация (`BadRequestException` с массивом строк в `message`) → `message: 'Validation failed'` + `details[]` (плоский массив, без per-field);
   - `Prisma.PrismaClientKnownRequestError`: `P2002` → 409, `P2025` → 404, прочее → 500 (safety net — явные исключения в коде остаются);
   - неизвестное → 500 с generic `message`.

3. **Скрытие деталей в prod.** Для 5xx полная ошибка (включая stack и детали Prisma) пишется только в лог через `Logger` (nestjs-pino); в тело уходит generic `message` в любом окружении — stack/SQL не утекают.

4. **Фронт.** `apiFetch` (`apps/web/src/lib/api.ts`) парсит тело по контракту `ApiErrorBody`, отдавая чистый `message` в `ApiError`; при неуспехе парсинга — фолбэк на `res.text()`. Поле `details` остаётся в серверном теле, но на фронте не потребляется. Существующие формы продолжают читать `err.status`, обратная совместимость сохранена.

---

## Последствия

### Положительные

- Любая ошибка API имеет один предсказуемый shape — фронт парсит её единообразно.
- В prod нет утечки stack/SQL в теле ответа; полная диагностика остаётся в логах.
- Статус-коды существующих эндпоинтов не изменились.

### Отрицательные / компромиссы

- Поле-уровневых ошибок валидации с бэка нет — точность подсветки полей целиком на client-side валидации (`react-hook-form + zod`).
- Маппинг кодов Prisma (`P2002`/`P2025`) ведётся вручную — новые сценарии надо добавлять в фильтр явно.
