# ADR-009: Gate — слой интеграции с внешними сервисами

**Статус:** Принято
**Дата:** 2026-03-24

---

## Контекст

При интеграции с внешними HTTP-сервисами (сторонние API, очереди сообщений, внешние платёжные системы и т.д.) логика транспорта смешивается с бизнес-логикой: retry, rate-limit, парсинг HTTP-ошибок оказываются в тех же классах, что строят запросы или обрабатывают доменные данные. Это затрудняет тестирование и повторное использование транспортного кода.

---

## Решение

Ввести тип классов **Gate** — тонкая обёртка над одним внешним сервисом или API.

### Ответственность Gate

| Входит в Gate                                    | Не входит в Gate                          |
| ------------------------------------------------ | ----------------------------------------- |
| Инициализация SDK / HTTP-клиента                 | Бизнес-логика (построение запросов)       |
| Отправка запроса и получение сырого ответа       | Парсинг доменных объектов из ответа       |
| Retry при 429 / 503 с правильной задержкой       | Исключения NestJS (`BadRequestException`) |
| Логирование транспортных ошибок                  | Знание о доменных сущностях               |
| Проверка доступности (ключ API задан / не задан) |                                           |

Gate возвращает **сырой результат или null** — никогда не бросает доменных исключений.

### Соглашения по именованию

```
{Имя сервиса}Gate  →  StripeGate, TelegramGate, SmtpGate
```

Файл: `src/{модуль}/{имя}.gate.ts`

Gate является провайдером NestJS и регистрируется в `providers` своего модуля.

### Пример: StripeGate

```
PaymentsService  →  StripeGate  →  Stripe SDK  →  Stripe API
```

- `StripeGate` — транспорт: создать платёж, обработать 429, вернуть сырой ответ
- `PaymentsService` — логика: сформировать параметры, распарсить результат, вернуть доменный объект

### Граф зависимостей

```
OrdersService  →  PaymentsService  →  StripeGate  →  Stripe API
```

---

## Альтернативы

| Вариант                                          | Почему отклонён                                           |
| ------------------------------------------------ | --------------------------------------------------------- |
| Вся интеграция в сервисе                         | Смешивает транспорт с логикой, тяжело тестировать         |
| Отдельный NestJS-модуль на каждый внешний сервис | Избыточно для одного клиента без собственных контроллеров |
| Общий `HttpGateway` для всех внешних вызовов     | Теряет типизацию конкретного API, слишком абстрактно      |

---

## Последствия

**Плюсы:**

- Сервисы тестируются без реального HTTP — достаточно замокать Gate
- Retry / rate-limit логика сосредоточена в одном месте
- Замена SDK (например, смена платёжного провайдера) затрагивает только Gate

**Минусы:**

- Дополнительный файл и слой косвенности на каждую интеграцию
