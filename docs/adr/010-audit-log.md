# ADR-010: Аудит-лог действий пользователей

**Статус:** Принято
**Дата:** 2026-06-08

---

## Контекст

В шаблоне не было механизма аудита: невозможно узнать, кто и когда выполнял значимые действия (логин, смена роли, удаление сущностей). Для production-ready основы такой журнал нужен и для безопасности, и для разбора инцидентов.

Требования: фиксация значимых действий **без засорения бизнес-логики** ручными вызовами в каждом месте; расширяемость под новые типы событий; защита от утечки чувствительных данных (пароли, токены).

---

## Решение

Аудит реализован через **декларативный декоратор `@Audit(options)` на методах контроллеров + один глобальный `AuditInterceptor`**, читающий метаданные через `Reflector`.

```
HTTP-запрос
  -> Guard (JwtAuthGuard) проставляет req.user
  -> AuditInterceptor: читает @Audit-метаданные хендлера
       success -> tap(): пишет options.event (success=true)
       error   -> catchError(): пишет options.failureEvent ?? options.event (success=false), затем rethrow
  -> AuditService.record() (fire-and-forget) -> AuditRepository -> таблица audit_logs
```

Ключевые решения:

1. **Whitelist метаданных.** Interceptor никогда не логирует тело запроса целиком — в журнал попадают только поля, явно возвращённые резолверами `target`/`metadata`/`actor` из декоратора. Это гарантирует, что пароли/токены не утекут.
2. **`actorId`/`targetId` без FK на `User`** + денормализованный `actorEmail`. Журнал должен переживать удаление пользователя, поэтому каскадных связей нет.
3. **`metadata` как `Json`.** Структура события расширяема без миграций — легко добавлять новые типы.
4. **Тип события — Zod-схема `AuditEventSchema` в `@repo/types`** (source of truth). Prisma-enum `AuditEvent` зеркалит её (импорт из пакета в `schema.prisma` невозможен — значения сверяются вручную).
5. **Запись fire-and-forget.** Сбой аудита логируется и не ломает основной запрос.
6. **Доступ к журналу — только ADMIN** (`JwtAuthGuard` + `RolesGuard` + `@Roles(ADMIN)` на API; redirect не-ADMIN в `admin/layout.tsx` на web).
7. **IP актора через `trustProxy`.** Fastify создаётся с `trustProxy: true` (`main.ts`), чтобы за nginx `request.ip` брался из `X-Forwarded-For`, а не был IP прокси. При прямом публичном доступе без доверенного прокси значение `X-Forwarded-For` можно подделать — тогда стоит указать доверенный CIDR прокси вместо `true`.

Фиксируемые события: `LOGIN_SUCCESS`, `LOGIN_FAILED`, `LOGOUT`, `USER_ROLE_CHANGED`.

---

## Альтернативы

| Вариант                        | Почему отклонён                                                                            |
| ------------------------------ | ------------------------------------------------------------------------------------------ |
| Event-based (`EventEmitter`)   | Требует ручных `emit`-вызовов в бизнес-логике — противоречит требованию «без засорения»    |
| Чистый interceptor по роутам   | Маппинг «роут→событие» прячется от эндпоинта и хуже масштабируется при росте числа событий |
| Ручные вызовы `audit.record()` | Засоряют сервисы транспортной заботой, легко забыть про неуспешные ветки (например, login) |

---

## Последствия

**Плюсы:**

- Фиксация декларативна и видна рядом с эндпоинтом; бизнес-логика не затронута.
- Неуспешный логин ловится через `catchError` без явного кода в сервисе.
- Whitelist-резолверы исключают утечку чувствительных данных by design.
- Новый тип события добавляется без изменения механизма (см. гайд).

**Минусы:**

- Аудируются только действия, проходящие через HTTP-контроллер с декоратором (внутренние вызовы вне HTTP не покрываются).
- Дублирование значений enum между Prisma и Zod требует ручной сверки.

**Вне рамок (развитие):**

- Политика хранения/ротации старых записей (cron-очистка или партиционирование, `AUDIT_RETENTION_DAYS`).
- Обогащение `actorId` для `LOGIN_SUCCESS` (сейчас гарантирован только `actorEmail`).
